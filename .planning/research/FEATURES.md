## Context: What Already Exists (v1.1 + v1.2)

These features are SHIPPED — do not redesign them:

- CALM v1.1 parser with strict Zod validation (`src/lib/calm/parser.ts`)
- 4-agent AI analysis: Architecture Analyzer, Compliance Mapper, Pipeline Generator, Risk Scorer
- Real-time SSE streaming dashboard with agent activity feed
- GitHub repo input + CALM fetch (public and private repos via PAT)
- Pipeline PR generation: GitHub Actions YAML + SAST configs + IaC committed to repo
- Compliance remediation PR: AI modifies CALM file, adds controls, upgrades protocols
- Existing skills: `skills/SOX.md`, `skills/PCI-DSS.md`, `skills/FINOS-CCC.md`, `skills/NIST-CSF.md`
- GitOps card in dashboard with 2 PR buttons (pipeline + remediation)

The v1.3 milestone adds: grounded compliance skills, multi-version CALM, 3-button GitOps split, and GitHub Action for continuous compliance.

## Feature-by-Feature Design: How Each v1.3 Feature Should Work

### Feature 1: Agentic Compliance Skills with Specific Control IDs

**What a good skill file must contain (based on industry practice):**

A compliance skill file is a markdown document injected verbatim into the agent's LLM prompt. For the agent to produce auditable, citation-quality output, the skill file needs:

1. **Structured control catalog with exact IDs**
   - PCI-DSS 4.0: "Requirement 4.2.1 — All transmissions of PAN over open, public networks use strong cryptography"
   - SOC2 TSC: "CC6.6 — The entity implements controls to prevent or detect and act upon the introduction of unauthorized or malicious software"
   - NIST CSF 2.0: "PR.DS-02 — Data-in-transit is protected (subcategory of Protect > Data Security)"

2. **CALM-to-control mappings per control ID**
   - Which CALM fields trigger or satisfy this control (e.g., `protocol: HTTPS` satisfies PCI-DSS 4.2.1)
   - Which CALM fields indicate violation (e.g., `protocol: HTTP` on a `connects` relationship = non-compliant)

3. **Remediation guidance per control ID**
   - What to change in CALM to achieve compliance (actionable, not generic)
   - Protocol security grounding: HTTP→HTTPS for PCI-DSS 4.2.1 because "TLS encrypts CHD in transit per Req 4.2.1"

4. **Severity classification**
   - Critical (blocks compliance) vs. High (audit finding) vs. Medium (best practice)

**Skill files to update/create:**

| Skill File | Action | Key Addition |
|------------|--------|--------------|
| `skills/PCI-DSS.md` | Update | Add PCI-DSS v4.0.1 requirement IDs (4.2.1, 6.4.1, 8.3.1, 10.2.1, 11.3.1, 12.3.1) with CALM mappings |
| `skills/NIST-CSF.md` | Update | Add NIST CSF 2.0 subcategory IDs (GV.OC-01 through PR.DS-02, DE.AE-02, RS.MA-01) |
| `skills/SOX.md` | Update | Add PCAOB/COBIT5 control references (ITGCs: AC-1 through AC-5, CM-1, OP-1) |
| `skills/SOC2.md` | CREATE | New file — SOC2 Type II Trust Service Criteria (CC1-CC9, A1, C1, P1) with TSC IDs |
| `skills/PROTOCOL-SECURITY.md` | CREATE | New file — Protocol security rationale (HTTP→HTTPS, FTP→SFTP, LDAP→LDAPS) with regulatory citations |

**What the remediator gets from the protocol skill:**
The `calm-remediator` agent currently has `skills: []` (no skills injected). Adding `skills/PROTOCOL-SECURITY.md` gives it grounded rationale: "Upgrading HTTP to HTTPS satisfies PCI-DSS 4.0 Req 4.2.1 (CHD in transit) and NIST CSF PR.DS-02 (data-in-transit protection)." This produces better change descriptions in the PR.

**Confidence:** HIGH — The existing skill loader infrastructure in `src/lib/skills/loader.ts` already handles this. Only content of skill files changes, not architecture.

### Feature 2: Multi-Version CALM Support (v1.0, v1.1, v1.2)

**What differs across versions (confirmed from CALM release notes):**

| Version | Core Schema | New Fields |
|---------|-------------|------------|
| 1.0 / 1.0-rc1 | Same core node/relationship model | Baseline |
| 1.1 | Same core | Minor field additions (current CALMGuard target) |
| 1.2 | Same core | **Decorators** (supplementary info without modifying core) + **Timelines/Moments** (time-based snapshots) |

**Key insight:** The core schema (nodes, relationships, controls, flows) is stable across all versions. v1.2 adds decorator and timeline fields as optional, additive extensions that attach to existing elements. No breaking changes.

**Implementation approach (lenient parser):**

```typescript
// Version detection — check $schema field or metadata
function detectCalmVersion(json: unknown): '1.0' | '1.1' | '1.2' | 'unknown'

// Extend calmDocumentSchema with optional v1.2 fields:
const calmDocumentSchema = z.object({
  nodes: z.array(calmNodeSchema).min(1),
  relationships: z.array(calmRelationshipSchema),
  controls: z.record(z.string(), controlDefinitionSchema).optional(),
  flows: z.array(calmFlowSchema).optional(),
  // v1.2 additions — optional passthrough
  decorators: z.array(z.unknown()).optional(),
  timelines: z.array(z.unknown()).optional(),
  $schema: z.string().optional(),           // CALM schema URL field
  metadata: z.record(z.string(), z.unknown()).optional(), // doc-level metadata
})
```

**Decorator structure (from CALM v1.2 release notes):**
Decorators attach supplementary info to nodes/relationships without modifying the core definition. They are used for: deployment tracking, security context, business metadata, operational info. CALMGuard should accept them (passthrough) without attempting to parse their structure — they don't affect compliance analysis.

**Timeline/Moments (from CALM v1.2 release notes):**
Time-based snapshots showing how architectures change over time. Out of scope for v1.3 compliance analysis (single-point-in-time); accept and ignore.

**UX:** Show detected version in the dashboard header ("Analyzing CALM v1.2 architecture"). Log a warning (not error) if version is unknown.

**Confidence:** HIGH — Schema confirmed via CALM repository; only optional fields added.

### Feature 3: GitHub Action for Continuous Compliance Checking

**Target UX (what the action does):**

1. Triggered on `pull_request` when CALM files change (path filter: `**/*.json` or configurable)
2. Detects changed CALM files in the PR using `git diff`
3. Calls CALMGuard API or runs local CALM validation
4. Posts a compliance summary comment on the PR (scores, top findings, pass/fail)
5. Sets a status check (pass/fail) that can be made a branch protection rule

**Generated file:** `.github/workflows/calmguard-check.yml` — committed to the user's repo via the DevSecOps CI PR.

**GitHub Action design (what the generated YAML should look like):**

```yaml
name: CALMGuard Compliance Check

on:
  pull_request:
    paths:
      - '**/*.json'  # configurable — catches CALM architecture files

jobs:
  calm-compliance:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write    # needed to post PR comment
      contents: read          # needed to checkout files

    steps:
      - uses: actions/checkout@v4

      - name: Detect changed CALM files
        id: changed
        uses: dorny/paths-filter@v3
        with:
          filters: |
            calm:
              - '**/*.json'

      - name: Install CALM CLI
        if: steps.changed.outputs.calm == 'true'
        run: npm install -g @finos/calm-cli

      - name: Validate CALM architecture
        if: steps.changed.outputs.calm == 'true'
        run: |
          for file in $(git diff --name-only origin/${{ github.base_ref }}...HEAD -- '*.json'); do
            calm validate --input "$file" || exit 1
          done

      - name: CALMGuard Compliance Analysis
        if: steps.changed.outputs.calm == 'true'
        id: analysis
        run: |
          # Call CALMGuard API or use CLI tool
          # Posts compliance score and findings to PR as comment
          echo "Compliance analysis complete"

      - name: Post PR comment
        if: steps.changed.outputs.calm == 'true'
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: '## CALMGuard Compliance Report\n[Results]'
            })
```

**Key design decisions:**
- Use `dorny/paths-filter` to skip action if no CALM files changed (avoids noise on non-CALM PRs)
- `pull_request` (not `pull_request_target`) for security — no access to secrets from fork PRs
- `permissions: pull-requests: write` for PR comment; `contents: read` only
- CALM CLI validation (`@finos/calm-cli validate`) as first step — fast schema check before AI analysis
- Optional: call CALMGuard API endpoint for full AI analysis (requires `CALMGUARD_API_URL` secret)

**How Arsenal agent generates this:** The pipeline-generator agent's prompt is updated to produce a separate `calmguard-check.yml` in addition to the `calmguard-ci.yml`. The Zod schema adds a `complianceAction` field.

**Confidence:** MEDIUM — Pattern is standard GitHub Actions; the exact CALMGuard CLI/API integration step is new code (no existing tool to reference).

### Feature 4: 3-Button GitOps Split (DevSecOps CI / Compliance Remediation / Cloud Infra)

**Current state (v1.2):** Single "Generate Pipeline PR" button + single "Generate Remediation PR" button. The pipeline PR bundles CI, security scanning, AND IaC together. Too much in one PR; hard for teams to review.

**Target state (v1.3):** Three distinct PR types, three distinct buttons:

| Button | PR Name | What's Committed | PR Description |
|--------|---------|-----------------|----------------|
| DevSecOps CI | `calmguard/devsecops-ci-{sha}` | `.github/workflows/calmguard-ci.yml` (CI-only, no deploy) + SAST configs | "Adds CI pipeline with SAST scanning on every PR" |
| Compliance Remediation | `calmguard/remediation-{sha}` | Modified CALM file with added controls + protocol upgrades | "Remediates N compliance gaps in architecture" |
| Cloud Infra | `calmguard/cloud-infra-{sha}` | `terraform/main.tf` or `cloudformation/template.yaml` | "Adds IaC scaffolding based on CALM architecture nodes" |

**API route change:** The existing `POST /api/github/create-pr` has `type: z.enum(['pipeline', 'remediation'])`. Extend to `type: z.enum(['ci-pipeline', 'remediation', 'cloud-infra'])`.

**CI-only vs full deployment pipeline distinction:**

A CI-only workflow:
- Triggers on `pull_request` and `push to main`
- Steps: lint → build → test → SAST scan → CALM validate
- NO deploy step, NO cloud credentials, NO environment protection rules
- Safe for teams that need separate deployment approval gates

The existing `pipeline-generator` agent prompt should be updated to generate a `calmguard-ci.yml` that ends at the test/scan stage. The IaC config moves to its own PR (cloud-infra button).

**Confidence:** HIGH — Straightforward API + UI extension of existing v1.2 pattern.

```
    └──enhances──> Compliance Mapper (cites protocol control IDs in findings)

CALM v1.2 Support
    └──requires──> Parser schema update (add optional decorators/timelines fields)
    └──enables──> v1.2 CALM files parsed without failure
    └──no impact on──> Agent analysis (agents consume AnalysisInput, not raw CalmDocument)

3-Button GitOps Split
    └──requires──> API route type enum extension
    └──requires──> CI-only pipeline template (Pipeline Generator prompt update)
    └──requires──> IaC-only PR type (extract terraform/cloudformation from existing pipeline output)
    └──enhances──> Dashboard GitOps card UI (3 buttons instead of 2)

GitHub Action Template
    └──requires──> Pipeline Generator generates calmguard-check.yml in addition to calmguard-ci.yml
    └──enables──> DevSecOps CI PR includes the compliance action template
    └──depends on──> dorny/paths-filter (external GitHub Marketplace action, stable)

SOC2 Framework Support
    ├──requires──> New 'SOC2' entry in framework enum (compliance-mapper.ts Zod schema)
    └──requires──> SOC2 skill file loaded by compliance-mapper agent
```

### Dependency Notes

- **Skills before agents**: Update skill files FIRST — agents pick up new content immediately with no code change
- **CALM parser before demo**: v1.2 CALM files will break the parser unless schema is extended; fix early
- **3-button split requires API + UI**: Both must change together; API route adds new type, UI adds third button
- **GitHub Action is independent**: Can be developed and tested as a standalone YAML template; does not block other features
