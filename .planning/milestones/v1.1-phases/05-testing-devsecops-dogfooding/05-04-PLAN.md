Create `SECURITY.md` at project root with startup-practical tone (per locked decision: clear, honest about scope, professional but authentic).

Structure:
1. **Security Overview** — Brief statement about CALMGuard's security posture. Acknowledge it's a hackathon project with production-grade security awareness.

2. **Vulnerability Reporting** — Contact email, expected response time (48h for hackathon context), what to include in a report (description, reproduction steps, impact assessment). Note: no bug bounty program.

3. **Security Practices**:
   - All CALM input validated via Zod schemas before processing
   - LLM outputs validated via Zod `generateObject` schemas (no free-form text in agent results)
   - No user data persistence (read-only analysis tool)
   - HTTPS enforced in production (Vercel)
   - Dependency scanning in CI (pnpm audit + license-checker)
   - SAST scanning (CodeQL + Semgrep)
   - No secrets in client-side code (API keys server-side only)

4. **Threat Model** (per locked decision: CALMGuard-specific actual attack surface):

   **Threat 1: Malicious CALM JSON (Input Injection)**
   - Surface: POST /api/analyze, POST /api/calm/parse
   - Attack: Crafted CALM JSON with oversized strings, deeply nested structures, Unicode sequences
   - Mitigation: Zod schema strict validation, bounded arrays, no file system writes from CALM data
   - Residual risk: DoS via very large valid documents (no rate limiting in hackathon version)

   **Threat 2: LLM Prompt Injection via Architecture Descriptions**
   - Surface: Agent prompts constructed from CALM node names, descriptions, control fields
   - Attack: Prompt injection in CALM description fields to manipulate agent outputs
   - Mitigation: `generateObject` with strict Zod output schemas — LLM output parsed regardless of injection. Structured output mode limits attack surface.
   - Residual risk: Agent findings may be misleading if LLM successfully manipulated. Recommend human review.

   **Threat 3: SSE Stream Tampering**
   - Surface: SSE stream between server and client
   - Attack: MITM injection of false agent events (fabricated compliance scores)
   - Mitigation: HTTPS (Vercel TLS), same-origin SSE, read-only tool (no state mutations from SSE events)
   - Note: Limited blast radius — no persistent data modified by SSE events.

5. **Dependencies** — Statement about dependency hygiene: automated audit in CI, license compliance checks for financial services requirements.

Tone: Be direct, not corporate. Use "we" not "the organization." Acknowledge scope limitations honestly.

**Pre-commit hooks (DSOP-05):**

Install husky and lint-staged:
```bash
pnpm add -D husky lint-staged
pnpm exec husky init
```

This creates `.husky/pre-commit`. Update it to contain:
```bash
pnpm exec lint-staged
```

Add `lint-staged` config to `package.json`:
```json
{
  "lint-staged": {
    "*.{ts,tsx}": [
      "eslint --fix --max-warnings=0"
    ]
  }
}
```

Note: Intentionally NOT including `tsc --noEmit` in lint-staged — typecheck always checks the whole project and would be slow. Typecheck runs in CI instead. ESLint --fix is sufficient for pre-commit gate.

Add `"prepare": "husky"` to package.json scripts (husky init should do this automatically, verify).

**Branch protection (DSOP-06):**
