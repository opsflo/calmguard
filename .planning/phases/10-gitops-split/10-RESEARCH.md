## Summary

Phase 10 splits the existing two-button GitOps card (Pipeline PR + Remediation PR) into three distinct PR buttons targeting different review audiences. The architecture is already well-suited for this — the `create-pr` API route already handles two PR types via a `type` discriminator, and the Zustand store already tracks `pipelinePR` and `remediationPR` as `PRRecord` objects. The work is a surgical expansion: add a third `infraPR` state, extend the `PRRecord` type union, add a new `type: 'infra'` branch to the API route, and refactor the GitOpsCard UI from 2 columns to 3 buttons in a row.

The Cloud Infrastructure generation is the only genuinely new agent capability. Arsenal (pipeline-generator) already produces a `infrastructureAsCode.config` Terraform snippet, but it is bundled with CI workflow and scanning configs in a single pipeline PR. The split requires Arsenal to generate a separate, standalone AWS Terraform scaffold using a new `CLOUD-INFRASTRUCTURE.md` skill. The CALM-to-AWS signal mapping is well-defined in the user decisions: database nodes → RDS + security groups, service nodes → ECS/ALB, HTTPS relationships → ACM + TLS listeners, mTLS → AWS Private CA.

Label creation via the GitHub Labels API (POST /repos/{owner}/{repo}/labels, 422 if already exists) is the only new GitHub API surface. The pattern fits cleanly into the existing `githubFetch`/`operations.ts` pattern. The concurrency guard (disable all three buttons while any one generates) requires a small amount of new Zustand state — a `generatingAny` derived selector or a `lockCount` counter is simpler than per-button mutexes.

**Primary recommendation:** Single API route (`/api/github/create-pr`) expanded with `type: z.enum(['pipeline', 'remediation', 'infra'])`, not three separate routes. This minimizes duplication of the SSE streaming boilerplate, GitHub error mapping, and token guard.

### Pattern 6: Label creation via GitHub API

The GitHub Labels API returns 422 when a label already exists. The safe pattern is try-create, ignore-422:

```typescript
// In src/lib/github/operations.ts
export async function ensureLabel(
  owner: string,
  repo: string,
  name: string,
  color: string,
  description: string,
  token: string,
): Promise<void> {
  const res = await githubFetch(`/repos/${owner}/${repo}/labels`, {
    method: 'POST',
    token,
    body: JSON.stringify({ name, color, description }),
  });
  // 201 = created, 422 = already exists — both are fine
  if (!res.ok && res.status !== 422) {
    const body = await res.text();
    throw new Error(`Failed to ensure label "${name}" (${res.status}): ${body}`);
  }
}

export async function addLabelToPR(
  owner: string,
  repo: string,
  prNumber: number,
  labels: string[],
  token: string,
): Promise<void> {
  const res = await githubFetch(`/repos/${owner}/${repo}/issues/${prNumber}/labels`, {
    method: 'POST',
    token,
    body: JSON.stringify({ labels }),
  });
  if (!res.ok) {
    // Label failures are non-critical — log but don't throw
    console.warn(`Failed to add labels to PR #${prNumber}: ${res.status}`); // nosemgrep
  }
}
```

### Anti-Patterns to Avoid

- **Separate API routes per PR type**: Creates three copies of the SSE streaming boilerplate, GitHub error mapping, and token guard. Use the single route with `type` discriminator.
- **Client-side concurrency lock**: Don't use `useState` for the lock — derive `isAnyGenerating` from the three Zustand PRRecord statuses. Single source of truth.
- **Overwriting existing PRSection subcomponent**: PRSection already handles all states correctly. The only change is passing `onGenerate={undefined}` to disable (already supported via the `isComingSoon` path).
- **Throwing on label failures**: Labels are cosmetic — don't block PR creation if label creation fails. Use `console.warn` and continue.
- **Multi-file Terraform for Cloud Infra**: Use the `modules` array within a single `generateObject` call (schema above). The API route commits each module as a separate file path. This avoids multiple LLM calls per PR.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Concurrency mutex | Custom lock state | Zustand selector `isAnyGenerating` from three `.status` fields | Already in store, zero extra state |
| Label create-or-ensure | Custom retry loop | Try POST /labels, ignore 422 | GitHub API already returns 422 for duplicates |
| SSE streaming for infra branch | New ReadableStream boilerplate | Add `else if (type === 'infra')` block to existing stream factory | All boilerplate (encoder, emit fn, error handler) already exists |
| Terraform generation from scratch | Manual HCL string construction | `generateObject` with `cloudInfraConfigSchema` + `CLOUD-INFRASTRUCTURE.md` skill | LLM already writes CALM-grounded IaC in pipeline-generator |

## Common Pitfalls

### Pitfall 1: Branch Naming Change Breaks Existing Behavior
**What goes wrong:** Changing `calmguard/pipeline-{ts}` to `calmguard/devsecops-ci-{ts}` requires updating both the API route AND any tests that assert on branch name patterns.
**Why it happens:** The branch name is baked into the SSE `done` event and displayed in the UI as `record.branchName`.
**How to avoid:** Grep for `calmguard/pipeline` before changing. The UI just displays whatever `branchName` the server sends — no client-side changes needed for the rename.
**Warning signs:** Test failures asserting `branchName.includes('pipeline')`.

### Pitfall 2: `__lastCloudInfraResult` Not Set Before PR Button Click
**What goes wrong:** User clicks "Cloud Infrastructure" before analysis has run (or Arsenal failed). The API route reads `globalThis.__lastCloudInfraResult` which is `undefined`.
**Why it happens:** The infra generator runs during analysis (same as pipeline-generator), but the global might be stale from a previous session in the same server process.
**How to avoid:** Gate the button behind `isComplete && githubRepo` (already done in dashboard). In the API route, throw a descriptive error: `'Cloud infrastructure analysis not available. Run analysis first.'`
**Warning signs:** Silent failures or confusing 500s when clicking Cloud Infrastructure before analysis.

### Pitfall 3: Grid Layout Overflow on Small Screens
**What goes wrong:** `grid-cols-3` at `md:` breakpoint puts three equal-width cards side by side; if card content is long, cards shrink below readability threshold.
**Why it happens:** Each PRSection has a title, description, and button — minimal content, but PR URLs can be long.
**How to avoid:** Use `truncate` on `branchName` display (already present in current PRSection). Use `grid-cols-1 md:grid-cols-3` to stack on mobile.
**Warning signs:** Horizontal scroll on medium-width viewports.

### Pitfall 4: PRSection `isComingSoon` Logic Conflict with Disable-All
**What goes wrong:** Current PRSection uses `onGenerate === undefined` to detect "coming soon" state and renders a disabled "Coming Soon" button. The disable-all concurrency mechanism also passes `undefined` — this would incorrectly show "Coming Soon" for active buttons during generation.
**Why it happens:** Overloading `undefined` for two semantics: coming-soon and temporarily-disabled.
**How to avoid:** Change PRSection to accept an explicit `disabled` prop instead of relying on `onGenerate === undefined`. Or: check `record.status === 'generating'` in the parent to show the spinner (which it already does). The simplest fix: pass a no-op function `() => {}` instead of `undefined` when temporarily disabling, so the button renders enabled but does nothing. Better: add a `globallyDisabled: boolean` prop to PRSection.
**Warning signs:** Non-generating buttons showing "Coming Soon" during a sibling PR generation.

### Pitfall 5: Cloud Infra Agent Missing from Orchestrator Phase 1
**What goes wrong:** If `generateCloudInfra` is added to the orchestrator's `Promise.all` Phase 1 but the CALM analysis globals are needed in sequence (analysis result → cloud infra), it must run AFTER Phase 1.
**Why it happens:** Cloud infra generation needs the CALM document (available at analysis start) but benefits from compliance data (available after Phase 1).
**How to avoid:** Run `generateCloudInfra` in orchestrator Phase 1 (same as `generatePipeline`) — it only needs `AnalysisInput`, not compliance results. Store result in `globalThis.__lastCloudInfraResult` same as `__lastPipelineResult`. Do NOT wait for Phase 2 (risk scorer).
**Warning signs:** Long PR generation times because cloud infra waits for risk scorer.

### Pitfall 6: DevSecOps CI PR Still Contains IaC
**What goes wrong:** The existing `buildPipelineFiles()` function in the API route includes `terraform/main.tf` (or `cloudformation/template.yaml`). After the split, DevSecOps CI PR must contain ONLY CI workflow and scanning tool configs — no IaC.
**Why it happens:** Phase 7 bundled everything together. GOPS-02 explicitly requires "NO deployment stages" and no IaC.
**How to avoid:** Modify `buildPipelineFiles()` to exclude the IaC file. The Cloud Infrastructure PR now exclusively owns Terraform. Build a `buildInfraFiles()` function that takes `CloudInfraConfig` and produces the terraform module files.
**Warning signs:** GOPS-02 success criterion fails: "DevSecOps CI PR...with NO deployment stages."

## Open Questions

1. **Should `generateCloudInfra` run during the main analysis or on-demand when the PR button is clicked?**
   - What we know: Pipeline generator runs during analysis (Phase 1) and stores result in `__lastPipelineResult`. Cloud infra generation is similar — it only needs `AnalysisInput`.
   - What's unclear: Adding cloud infra to Phase 1 increases the analysis time by one more parallel LLM call. The analysis already takes 15-30s.
   - Recommendation: Run in Phase 1 alongside `generatePipeline`. The parallelism means it's "free" in wall-clock time. The result is pre-computed when the user clicks the button.

2. **Single Terraform file vs multi-file module organization for Cloud Infra PR?**
   - What we know: The schema above uses a `modules` array where each entry has a distinct `filename`. The API route commits each as a separate file.
   - What's unclear: LLM reliability generating multiple distinct HCL files in a single `generateObject` call.
   - Recommendation: Start with single file (`terraform/main.tf`) for reliability; use modules array with one entry. If planner wants multi-file, the schema already supports it — just generate multiple entries.

3. **Sonner toast for error vs inline error in PRSection?**
   - What we know: Current PRSection shows inline error state with retry button. The context decisions mention "error toast notification" on failure.
   - Recommendation: Show both — inline error state (already implemented) + `toast.error()` via Sonner (already installed in the project). Non-breaking addition.
