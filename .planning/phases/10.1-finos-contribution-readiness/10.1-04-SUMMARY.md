## What Was Built

### 6 Architecture Decision Records (ADRs)

Each ADR follows MADR format (Status, Date, Deciders, Context, Decision, Consequences) and is ~60-80 lines. Written in factual, grounded language targeting a FINOS technical reviewer:

- **ADR-001:** YAML files for agent config — YAML chosen over JSON (no comments) and TypeScript constants (requires recompile). `yaml` package already present.
- **ADR-002:** Zod for runtime validation — single source of truth for TS types and runtime safety; required by Vercel AI SDK `generateObject`.
- **ADR-003:** localStorage for compliance learning store — hackathon-scoped decision with explicit PostgreSQL migration path (4-step, 2-3 day estimate).
- **ADR-004:** Promise.allSettled for Phase 1 orchestration — graceful degradation vs Promise.all (one failure cancels all); 4x wall-clock reduction over sequential.
- **ADR-005:** Multi-provider LLM registry — Vercel AI SDK abstraction allowing provider swap via env var, no agent rewrites required.
- **ADR-006:** Markdown skill files — domain experts can edit compliance knowledge without TypeScript; tables natural for control ID matrices.

### REVIEWING.md (Repo Root)

Ordered 8-step reading path for Eddie:
1. Agent configuration format (agents/*.yaml)
2. Core type system (agents/types.ts, calm/types.ts)
3. Orchestration flow (orchestrator.ts, registry.ts)
4. Full agent walkthrough (compliance-mapper end-to-end)
5. CALM parsing (parser.ts, normalizer.ts)
6. Compliance learning engine (store.ts, extractor.ts)
7. Run the tests (pnpm test:run, 16 suites)
8. Architecture Decision Records

Includes code-size context table scoping reviewer to the ~4,100 lines of core logic (not the ~4,800 lines of generated shadcn/ui boilerplate). Includes "What to Look For" checklist (license headers, no globalThis, no `any`, supply chain).
