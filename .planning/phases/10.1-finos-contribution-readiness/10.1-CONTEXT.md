## Implementation Decisions

### Item 1: SPDX License Headers (MUST FIX — ~30 min)
- Add to every `.ts`, `.tsx`, `.yaml` (agents/), and `.md` (skills/) file
- Header format (2 lines):
  ```
  // SPDX-License-Identifier: Apache-2.0
  // Copyright 2026 FINOS
  ```
- For YAML files use `#` comments; for Markdown use HTML comment or front-matter
- Script approach: bash script prepending headers to all 114 files in one pass
- Do NOT add to config files (package.json, tsconfig.json, .env, etc.)

### Item 2: Replace globalThis with Session Store (~2-3 hrs)
- There are 5 globalThis variables in orchestrator.ts and API routes sharing state between requests
- Replace with an in-memory session store keyed by request ID (UUID)
- Store lives in `src/lib/session-store.ts` or similar
- Must be safe for concurrent usage (multiple simultaneous analyses)
- Session cleanup: TTL-based or explicit cleanup after stream ends
- No external dependencies — pure in-memory Map

### Item 3: Expand CONTRIBUTING.md (~1 hr)
- Current state: 3 sentences (hackathon template)
- Content already exists in `docs/docs/contributing.md` — largely copy-paste
- Must include:
  - Dev setup instructions (pnpm, env vars, GOOGLE_GENERATIVE_AI_API_KEY)
  - Branching/commit conventions (Conventional Commits)
  - Testing guidelines (pnpm test:run, Vitest)
  - DCO sign-off examples (`git commit -s`)
  - How to add new agents, skills, compliance frameworks

### Item 4: React Component Tests (~3-4 hrs)
- Current: 13 test files, all backend logic, zero UI tests
- Add 3-5 snapshot tests minimum using @testing-library/react
- Target components: compliance-card, findings-table, risk-heat-map
- Use existing Vitest + jsdom setup
- Tests should render components and snapshot — not complex interaction tests
- Install @testing-library/react if not already present

### Item 5: Architecture Decision Records (ADRs) (~1-2 hrs)
- Location: `docs/adrs/` directory
- Format: short (1 page each), structured (Context → Decision → Consequences)
- 6 required ADRs:
  - ADR-001: Why YAML for agent configuration (vs. hardcoded TypeScript)
  - ADR-002: Why Zod for runtime validation (vs. TypeScript-only types)
  - ADR-003: Why localStorage for learning store (vs. database) + migration path to PostgreSQL
  - ADR-004: Why Promise.allSettled for Phase 1 orchestration (graceful degradation)
  - ADR-005: Why multi-provider LLM registry (vs. single provider)
  - ADR-006: Why skill files are markdown (vs. structured config)
- Use MADR (Markdown ADR) format or simple header structure
